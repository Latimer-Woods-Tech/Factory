## Operator decision tree (per IR playbook §3 + §5)

### Tripwire 1 — spam complaint

1. **Investigate the complainer.** Pull from Resend dashboard: the recipient signed up 4 months ago via a free chart calc landing page; opened 11 of 14 prior emails; reported THIS one as spam. Profile: legitimate signup, real complaint.
2. **Investigate the content.** Diff the Substack-promo body vs prior 5 sends — the promo had a "[Partner] just launched X, try it" embed that the recipient hadn't opted in to. Likely cause: ad-like content in an editorial sequence.
3. **Provider reputation check.** Resend dashboard shows reputation still "good." Complaint rate window still recoverable.
4. **Decision (per IR playbook §3 row 2):**
   - Content remediated: future cross-promos require explicit opt-in, not implicit inclusion.
   - 1% canary restart of `consumer-newsletter-04` for 7d.
   - `/approve` the issue with a comment specifying: "approved for 1% canary; full resume gated on T+7d clean."
5. **Time spent:** ~20 min including writing the canary plan.

### Tripwire 2 — unsub spike

1. **Investigate the cross-promo.** ExperimentRunner had projected 35% list overlap based on the partner's audience-overlap signal; actual was 22%. The 78% non-overlap mass-unsubbed within 12 h.
2. **Is this a real list-health failure?** No — the system worked: people who shouldn't be on the list left. It's a *targeting* failure (we shouldn't have sent to non-overlap), not a content failure.
3. **Threshold review:** The 2× rolling-14d median is correct for catching *content* failures. For *targeting* failures it produces a false positive — the same content sent to the right list would not have spiked. Does this argue for revising the threshold? **No** (the system did the right thing by pausing; the gap is not in the threshold). It argues for *better pre-flight list-match scoring* before the send is queued.
4. **Decision:**
   - Mark this issue `decision/threshold-correct, root-cause-targeting`.
   - File a sub-issue against the supervisor: "pre-flight list-match score required for cross-promo sends." Tier-2 backlog.
   - Close this tripwire issue with `/approve` — no content change required, no canary needed.
   - Resume `consumer-newsletter-04` once tripwire 1's canary completes.
5. **Time spent:** ~15 min.

### Tripwire 3 — brand-safety / image-voice critical

This is the heavy one.

1. **The practitioner.** Read the practitioner's onboarding history. Hand-onboarded 11 d ago, signed the design-partner agreement which includes a "no efficacy claims for regulated conditions" clause. They published in good faith, didn't realize the phrasing was over the line.
2. **The 247 viewers.** Most concerning — the shareable was *public* for 14h. ⚠️ See Gap 9 below. Decisions:
   - **No PII captured.** The library tracks views by IP-hash for rate limiting only; no identified user list to notify.
   - **Anonymous viewers do not trigger a notification obligation** under GDPR Art. 14 (no controller-subject relationship for an anonymous public-page view) or CCPA (no "sale of personal information"). Operator records this analysis in the issue.
   - **One known interaction:** the X-share. Operator does **not** post a public correction (per [`marketing-incident-response.md §5`](../../runbooks/marketing-incident-response.md#5-brand-crisis--viral-negative-mention) — "the autonomous loop never publishes a response to a negative mention; all responses are tier-3 artefacts drafted at operator request"). The mention isn't viral (1 share, 0 retweets, 0 quote-tweets). Decision: do nothing, document in the issue.
3. **The practitioner conversation.** Operator drafts a direct reply (not autonomous; operator-authored) to the practitioner: explains what the image-voice gate flagged, asks them to choose between (a) re-publish with rephrased body, (b) keep paused, (c) withdraw the reading entirely. Tier-3 drafting per playbook.
4. **The image-voice gate.** Was this a generator drift, a corpus drift, or a never-anticipated case? Operator inspects the gate's denylist — the regulated-health phrase IS in the denylist embedding set, but only as a `cypher_practitioner` rule, not a `selfprime:practitioner` rule. The CLIP step caught it via cross-cell embedding leakage. Lucky.
5. **Decision:**
   - Ship a config-only PR: hoist regulated-vertical phrases into a *shared* `universal_denylist` block in [`packages/validation/`](../../packages/validation/), referenced by every cell's image-voice rule set. Same-day tightening per [`CONSTITUTION.md §11`](../CONSTITUTION.md#11-amendment-process).
   - Ship ADR `2026-05-18-universal-regulated-denylist.md` (constitution amendment to §7).
   - Image-voice gate frequency: every 4h → **every 30 min** for the next 30d while the gap is monitored. Cost impact analyzed in §"Total system cost" below.
   - Practitioner-published path: add a *publish-time* (synchronous) image-voice block, not just retro batch. ⚠️ See Gap 3.
   - `/approve` the issue only after the config PR merges + the universal denylist deploys. Restart of `selfprime:practitioner` cell publication gated on the merge.
6. **Time spent:** ~90 min including PR + ADR drafting.

**Total operator time triaging the night:** ~2 h 5 min — well past the 5-min daily checklist. The night was an M-SEV-2 cluster, not the routine 5-min day.

## Integration gaps surfaced

> The most important section. Each gap is a real bug. ⚠️ markers correspond to inline markers above. Severity is the operator's call; these are this scenario's recommendations.

### ⚠️ Gap 1 — `selfprime:practitioner` shouldn't have published the phrase at all (cell-leakage)

**Surface:** image-voice gate denylist is per-cell, not universal. A regulated-health-claim phrase indexed under `cypher_practitioner` caught this reading *only by accident* (CLIP embedding similarity across cells). The supervisor has no notion of "universal denylist."

**Severity:** **High.** A different phrase that didn't share CLIP embeddings with cypher's denylist would have shipped silently.

**Fix:** ADR `2026-05-19-universal-regulated-denylist.md` (drafted in operator response). Hoist regulated-vertical phrases (health claims, financial advice, legal claims, claims about minors) into `packages/validation/src/universal-denylist.ts`. Every cell's rule set imports it.

**Owner:** operator + [`packages/validation/`](../../packages/validation/) maintainer.

### ⚠️ Gap 2 — image-voice batch frequency is wrong for the threat model

**Surface:** [`IMAGE_GENERATION.md §5`](../IMAGE_GENERATION.md#5-image-voice-gate) specifies the gate but is silent on cadence. Implementation defaulted to 4h batches. **A 14h exposure window is unacceptable** for any cell whose cell type (`selfprime:practitioner`) accepts user-generated content.

**Severity:** **High — Critical** for any user-generated-content path.

**Fix:** Two parts:
1. *Publish-time synchronous gate* on user-generated content (practitioner-published readings). The publish path waits for the gate; a critical/major hit blocks publication entirely, never serves the URL.
2. *Retro batch* for system-generated content drops from 4h to 30 min cadence for 30d, then reassess.

**Cost impact:** synchronous gate adds ~$0.003 per published reading (Claude vision batched-of-one). Practitioner publish rate ~2/day → +$0.006/day. 30-min batch cadence costs ~$0.20/day more.

**Owner:** [`packages/validation/`](../../packages/validation/) + supervisor Worker. ADR `2026-05-19-image-voice-publish-time-block.md`.

### ⚠️ Gap 3 — the 247 viewers and the "obligation to notify" question

**Surface:** [`DSR_HANDLING.md`](../DSR_HANDLING.md) handles *requested* erasure, not *operator-initiated retroactive removal*. The scenario forced operator to determine notification obligation in real time. The answer happened to be "no, viewers were anonymous," but no doc said so.

**Severity:** **Medium.** A future case where viewers *are* identified (e.g. a reading shared to a registered-user feed) has no playbook.

**Fix:** Add a §9 "operator-initiated retroactive removal" subsection to [`DSR_HANDLING.md`](../DSR_HANDLING.md) covering: anonymous-viewer no-notify rationale, identified-viewer notify procedure, audit row schema, search-console removal protocol, the 410-vs-restore-reactivation playbook (Gap 7).

**Owner:** @adrper79-dot + counsel review.

### ⚠️ Gap 4 — no false-positive learning loop for tripwires

**Surface:** Tripwire 2 (unsub spike) fired correctly, but the *root cause* was targeting, not list health. The system has no mechanism to label a tripwire fire as "true positive" / "false positive — root cause elsewhere" so future runs of the same evaluator can avoid the same false branch.

**Severity:** **Medium.** Cumulative false positives will desensitize the operator over time. Today: 1 false positive in 38 days. At 1/week it becomes alarm fatigue.

**Fix:** Add a `tripwire_outcomes` table that records `(tripwire_name, fire_id, operator_classification, root_cause_tag, threshold_correct_yn)`. RetroWriter consumes it weekly. If the same `(tripwire, root_cause)` tuple has >3 false-positives in 30d, RetroWriter opens an ADR draft proposing a threshold or scope revision.

**Owner:** TripwireMonitor + RetroWriter agents.

### ⚠️ Gap 5 — multi-tripwire clustering: 3 issues, no aggregate view

**Surface:** Tonight produced 3 tier-3 issues on overlapping cells within 38 min. The operator has no aggregate "incident night" view — each is a separate issue, separate notification, separate digest line. The digest's "max 3 escalations" rule conveniently fit this case but masks the *cluster* signal.

**Severity:** **Medium.** Today fits in 3 lines. A future 5-tripwire night exceeds the digest cap and one drops silently.

**Fix:** Two changes:
1. DigestComposer adds an "incident night detected" line when ≥3 tripwires fire in 12h, regardless of cell.
2. Open a single "incident night" umbrella issue with `kind/incident-night` that links the child tripwire issues. Operator can close the umbrella with `/approve-cluster` to bulk-close once each child is approved.

**Owner:** DigestComposer + supervisor Worker.

### ⚠️ Gap 6 — Pushover priority/DND interaction undefined

**Surface:** Three notifications fired between 23:30 and 06:00 to a sleeping operator. P1 doesn't break iOS DND by default; only P2 (emergency, requires ack, retries every 5 min) does. The current `tier:3` → P1 mapping in [`marketing-incident-response.md §8`](../../runbooks/marketing-incident-response.md#8-communication-channels) means **no tier-3 marketing event ever pages the operator at night.**

**Question this raises:** is that *correct*? Is autonomous marketing acceptable to be "operator catches up at 07:00"? The constitution says tier-3 wakes the operator; the IR playbook says priority-1 for tripwire fires. P1 + iOS DND = silence. Today's incident: 14h public exposure of a regulated claim before pause. *That* is the kind of thing that should wake an operator.

**Severity:** **High.** This is a misalignment between the IR playbook's priority scheme and operator on-call expectations.

**Fix:** Decide deliberately. Three options:
1. Promote brand-safety + legal-risk tripwires from P1 to P2 (wake on critical content); leave list-health + cost tripwires at P1.
2. Introduce an "operator-asleep window" (22:00 → 06:00 local) where only specific tripwires fire P2.
3. Accept the current state and document explicitly that "operator catches up at 07:00 Pushover digest" is the on-call expectation.

Operator preference recommended in retro: option 1, scoped to brand-safety + legal-hold + suspension-in-progress (M-SEV-1 trio).

**Owner:** [`marketing-incident-response.md`](../../runbooks/marketing-incident-response.md) §8.

### ⚠️ Gap 7 — paused shareable + downstream caches (X / Reddit preview)

**Surface:** CDN purge handles Cloudflare. The X share's OG image was cached by X (twitter.com) at share time; X holds the preview for ~24h. During that window, the X share renders the offending image *even though the slug serves 410*. No mitigation in place.

**Severity:** **Medium.** Reach is bounded (1 share). For a viral share this would be high.

**Fix:**
1. Add platform-specific cache-bust calls on tier-3 brand-safety pauses (X's `/i/buffer/...` not supported by API — practical mitigation is contact the share's poster to delete; document as operator workflow step).
2. Cache the originally-published image in R2 with a `compromised` marker so a future enumeration can flag any platform-cached previews.

**Owner:** [`marketing-incident-response.md §3 row 7`](../../runbooks/marketing-incident-response.md#3-the-7-brand-safety-tripwires) (image-voice block restart procedure).

### ⚠️ Gap 8 — self-referential voice-gate risk on system notification copy

**Surface:** The practitioner-pause email is LLM-generated and passes the voice gate. If the gate had failed — for instance, if a future variation reused some of the offending phrase as a "to be clear, we removed this language..." quote — the notification would have queued a tier-3 escalation for *the notification itself*. The practitioner would remain uninformed. No fallback to a static template exists.

**Severity:** **Medium.** Today's case worked because the template explicitly avoids the offending phrase. A more naïve template could deadlock.

**Fix:** Every "tripwire response" agent path must have a **static fallback template** stored in [`packages/copy/src/fallbacks/`](../../../packages/copy/) — plain English, no LLM, used when the LLM-generated version fails the voice gate. The fallback is gate-exempt because it was operator-pre-approved at registration time.

**Owner:** [`packages/copy/`](../../../packages/copy/) + supervisor Worker.

### ⚠️ Gap 9 — `selfprime:practitioner` cell config did not flag "publishes-public-content"

**Surface:** The supervisor treats `selfprime:practitioner` as a normal cell. It did not know that this cell uniquely exposes a *user-generated-public-content* surface, which is the only path in the whole loop that bypasses publish-time gate checks. There is no `cell_capabilities: ['publishes_user_content']` flag anywhere in [`ICP_MATRIX.md`](../ICP_MATRIX.md) or the cell config.

**Severity:** **Medium-High.** Future cells with similar surfaces (e.g. xicocity_creator if drops include user-uploaded media) will inherit the same blind spot.

**Fix:** Add a `capabilities` block to each cell's ICP file. Cells with `publishes_user_content: true` trigger publish-time gate enforcement automatically. Encode in `ICP_MATRIX.md` schema next revision.

**Owner:** [`ICP_MATRIX.md`](../ICP_MATRIX.md) + cell ICP files.
